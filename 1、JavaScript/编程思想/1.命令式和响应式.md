# 命令式 vs 响应式

1. 命令式：你一步一步告诉程序「怎么做」
2. 响应式：你声明「我要什么状态 / 关系」，一旦依赖变化，系统自动帮你更新

## 1. 命令式编程在前端里的典型样子

在原生 JS / jQuery 时代，我们基本都是 命令式：
核心思路：

- 手动获取 DOM 节点
- 手动在合适的时机读取状态
- 手动把最新状态「写回」到 DOM

```js
// 初始渲染
const countSpan = document.querySelector('#count');
const addBtn = document.querySelector('#add');
let count = 0;
countSpan.textContent = count;
// 事件里手动更新
addBtn.addEventListener('click', () => {
  count = count + 1;
  countSpan.textContent = count;      // 命令式更新视图
});
```

代码特点：  
- 你要 显式写出每一步 DOM 更新逻辑  
- 越复杂的 UI，越多 querySelector、innerHTML、textContent、classList.add/remove  
- 数据和视图之间的对应关系，全靠你脑子里维护  
优点：
- 控制力强：要改哪一块 DOM，你就直接操作它
- 简单场景里，上手快、理解直观

缺点（前端项目一大就会很明显）：

- 容易出现「忘了改某处 DOM」导致 UI 不一致
- 业务逻辑和 DOM 操作混在一起，难以重构、难以复用
- 状态多时，很难保证「所有使用这个状态的地方都同步更新」

## 2. 响应式（Reactive）在前端里的含义

响应式更关注「声明状态关系」，而不是「一步步命令怎么更新」。

核心理念：

- 有一个或多个「源数据（state）」
- 你声明：某些 UI / 派生数据依赖这些 state
- 当 state 改变时，框架自动计算「有哪些依赖它的地方」并更新它们

### 2.1 React 的响应式思路（函数式 UI）

以 React 为例（Hooks 时代）：

```js
function Counter() {
  const [count, setCount] = useState(0);   // 声明 state
  return (
    <div>
      <span>{count}</span>                 {/* 声明 UI 依赖 count */}
      <button onClick={() => setCount(count + 1)}>
        +1
      </button>
    </div>
  );
}
```

你做了什么：  
- 声明：UI = f(state)  
- 调用 setCount 时，只是改变 state  

React 做了什么：  
- 记录哪些地方依赖 count  
- setCount 后，自动重新渲染组件，生成新的虚拟 DOM  
- 找出 DOM 需要变的地方，打补丁更新  

你的关注点从「怎么更新 DOM」变成：

- 只管维护好状态
- 只管描述状态 → 视图的映射
这就是典型的「响应式思维」。
