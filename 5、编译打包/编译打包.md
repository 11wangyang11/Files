我们都知道，无论是nodejs项目还是web端的Javascript项目，都需要编译成对应的版本和模块，来适配其最终的运行环境。通常，我们是通过配置`tsconfig.json`文件以及Babel等编译工具。

# 一、tsconfig
```ts
// tsconfig.json文件
{
  "compilerOptions": {
    "target": "es6", // javascript版本
    "module": "CommonJS", // 模块系统
    "lib": ["ES2021"], // 要包含ES2021库
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  },
  "include": ["src/**/*"], // 编译的文件
  "exclude": ["node_modules", "**/*.spec.ts"] // 不编译的文件(优先级高于include)
}
```
上面是某Node.js项目中的`tsconfig.json`文件。这里针对`tsconfig.json`文件的“target”和“module”，详细介绍一下。
## 1、配置介绍
### （1）target
`target`指的是代码编译后的javascript版本，比如ES6支持箭头函数、let 和 const 声明等，这意味着编译后的代码需要在支持 ES6 的环境中运行。这里要注意“是编译后”，编译前的项目代码自然可以使用任何最新的javascript语法。 

既然如此，是不是`target`版本越低越好？这样就可以支持更多的js环境了。虽然设置较低的版本（如 ES3 或 ES5）可以使编译后的代码在更多的旧环境中运行，但这并不总是最佳选择。一方面，使用较低版本的 target 会导致编译器将现代语法转换为较旧的等效代码，这可能会使编译后的代码变得冗长且难以阅读。例如，箭头函数会被转换为传统的函数表达式，let 和 const 会被转换为 var，这可能会影响代码的可读性和调试体验。另一方面，较低版本的 target 可能不支持一些现代 JavaScript 特性，这些特性可能在性能和功能上有显著优势。例如，ES6 引入了 Promise、async/await、模块系统等，这些特性在现代 JavaScript 开发中非常有用。如果将 target 设置为较低版本，可能需要引入 polyfills 或其他替代方案来实现这些功能，这可能会增加代码的复杂性和体积。

### （2）module
`module`指定编译后的代码将使用哪种模块加载和组织方式。例如，CommonJS 使用 require 和 module.exports，而 ES6 模块系统使用 import 和 export。比如上述`tsconfig`配置（`"target": "es6","module": "CommonJS"`）针对下面代码：
```ts
const greet = (name: string) => {
  console.log(`Hello, ${name}!`);
};
export default greet;
```
编译后的 JavaScript 代码为：
```js
"use strict";
Object.defineProperty(exports, "__esModule", { value: true }); // 给exports增加__esModule属性，表示这个 CJS 包是从 ES Module 转出来的，默认导出在 exports.default 上
const greet = (name) => {
    console.log(`Hello, ${name}!`);
};
exports.default = greet;
```
所以，该文件导出的是`{default: greet, __esModule: true}`。在这个例子中，编译后的 JavaScript 代码使用了 ES6 的箭头函数和模板字符串（因为 target 设置为 ES6），同时使用了 CommonJS 的模块系统，如exports.default = greet;（因为 module 设置为 CommonJS）。如果使用`module: 'es6'`，则编译结果如下：
```js
const greet = (name) => {
    console.log(`Hello, ${name}!`);
};
export default greet;
```

**举例**
我们经常使用如下代码：
```js
const module = require('./module')
const Tag = module && module.__esModule ? module.default : module
export default Tag
```
其实这是CommonJS导入的两种场景，如果是CommonJS的`exports.default = greet;`导出(判断__esModule为true)，则使用`require('./module').default;`的方式导出；如果是CommonJS的`module.exports = someFunction;`的方式导出，则`require('./module')`就是导出的对象。但是如果是ES Module模块导出，则require会报错，所以使用上面方式导入，理论上项目的tsconfig.json的module得是CommonJS版本。但是，很奇怪的是我tsconfig.json配置的是`"module": "es2020"`，但是却没有报错，可以正常使用。这是为什么呢？

这归功于打包器了，项目使用了Metro/Webpack 这类打包器，打包器会在构建阶段把 import/export 与 require 统一成自己的模块系统，并注入 CJS/ESM 互操作（含 esModule）逻辑，所以 require(...) 拿到的 ESM 源也能正常用。只有没有打包器的时候才会报错。我们项目编译后的产物是dist，里面是esm导出，而metor打包后的产物本地看不到，需要主动处理。

### (3) jsx
在 tsconfig.json 文件中，jsx 选项用于指定 TypeScript 如何处理 JSX 语法。以下列举了 jsx 几种常用设置及其含义：
1. "preserve": 保留 JSX 语法，不进行转换。这意味着你需要一个后续的编译步骤（通常是 Babel）来处理这些 JSX 语法。
2. "react": 将 JSX 语法转换为 React.createElement 调用。适用于传统的 React 项目和 React-Native 项目，需要显式导入 React。
3. "react-jsx": 使用新的 JSX 转换机制，将 JSX 语法转换为 jsx 函数调用，不需要显式导入 React。适用于使用 React 17 及以上版本的项目，简化代码。

### (4) 其他重要配置项

#### (a) declaration 和 declarationMap
- `"declaration": true` — 生成 TypeScript 类型声明文件（.d.ts），这样其他项目导入你的库时可以获得类型提示。
- `"declarationMap": true` — 生成声明文件的源映射，用于调试时追踪到原始 TypeScript 代码。

这在发布 npm 包时非常重要。例如：
```json
{
  "compilerOptions": {
    "declaration": true,
    "declarationDir": "./dist/types",
    "declarationMap": true
  }
}
```

编译后会生成：
```
dist/
  index.js
  index.d.ts
  index.d.ts.map
```

#### (b) sourceMap
- `"sourceMap": true` — 生成源映射文件（.js.map），用于在浏览器或调试工具中调试编译后的代码时，能够映射回原始的 TypeScript 代码。
- `"inlineSourceMap": true` — 将源映射内联到编译后的 JavaScript 文件中，不生成单独的 .map 文件。

开发环境通常启用 sourceMap 以便调试，生产环境根据需要选择。

#### (c) noUnusedLocals 和 noUnusedParameters
- `"noUnusedLocals": true` — 检测未使用的局部变量，如果有会报错。
- `"noUnusedParameters": true` — 检测未使用的参数，如果有会报错。

这些配置帮助清理代码，避免存在无用的变量和参数。例如：
```ts
// 启用 noUnusedLocals 后，以下代码会报错
function greet(name: string) {
  const message = "Hello"; // 未使用，报错
  console.log(name);
}
```

#### (d) strict 及其子选项
- `"strict": true` — 启用所有严格的类型检查选项，包括：
  - `noImplicitAny` — 不允许隐含的 any 类型
  - `strictNullChecks` — 严格的 null 和 undefined 检查
  - `strictFunctionTypes` — 严格的函数类型检查
  - `strictBindCallApply` — 严格检查 bind、call、apply
  - `strictPropertyInitialization` — 类属性必须初始化

例如，启用 `strict` 后：
```ts
// 报错：参数 age 没有类型注解，隐含为 any
function setAge(age) {
  return age + 1;
}

// 正确做法
function setAge(age: number): number {
  return age + 1;
}
```

#### (e) baseUrl 和 paths
- `"baseUrl": "./"` — 设置模块解析的基础目录。
- `"paths": {}` — 定义模块路径映射，实现路径别名。

这在大型项目中非常有用，避免复杂的相对路径：
```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@components/*": ["src/components/*"],
      "@utils/*": ["src/utils/*"],
      "@types/*": ["src/types/*"]
    }
  }
}
```

然后可以这样导入：
```ts
// 而不是 import { Button } from '../../../components/Button'
import { Button } from '@components/Button';
import { formatDate } from '@utils/date';
```

#### (f) moduleResolution — 模块查找算法

**问题本质**：当你写 `import { foo } from 'bar'`，TypeScript 如何找到 `bar` 这个模块？

- `bar` 可能是 npm 包（要去 `node_modules` 找）
- `bar` 可能是相对路径（`./utils`）
- `bar` 可能是一个目录（要找到目录里的 `index.ts`）
- 还可能是 `bar/index.js`、`bar.ts`、`bar.d.ts`...

TypeScript 需要一套规则来判断按什么顺序去找，这就是 `moduleResolution`。

**两种模式的实际区别**：

```json
{
  "compilerOptions": {
    "moduleResolution": "node"  // Node.js 规则
  }
}
```

```typescript
// node_modules/foo/package.json
{
  "name": "foo",
  "main": "lib/index.js",      // Node.js 默认入口
  "exports": {                  // 现代条件导出
    "import": "./es/index.js",  // ESM 入口
    "require": "./lib/index.js" // CJS 入口
  }
}

// 你的代码
import foo from 'foo';
```

| 配置 | 查找过程 | 支持特性 |
|------|---------|--------|
| `node` | 只看 `main` 字段 | 传统 Node.js 方式，较简单 |
| `bundler`（推荐） | 解析 `exports` 条件导出 | 支持现代 npm 包格式，推荐用 |

**选择建议**：
- **Node.js 项目**：使用 `"moduleResolution": "node"`
- **前端项目 + 打包工具**：使用 `"moduleResolution": "bundler"`（TypeScript 5.0+）

#### (g) isolatedModules — 单文件编译兼容性

**问题背景**：TypeScript 编译器能进行完整的项目级类型分析，但 Babel、Metro 等工具是**单文件编译**（每次只处理一个文件，不知道其他文件）。

这导致一些 TypeScript 特性在单文件编译时无法正确处理：

```typescript
// a.ts
export enum Status {
  Pending = 0,
  Success = 1
}

// b.ts
import { Status } from './a';
console.log(Status.Success);  // 输出 1
```

**问题**：`enum` 在 TypeScript 中会被编译成 JavaScript 对象，但值的计算需要知道 `Status` 的完整定义。Babel 编译 `b.ts` 时，不知道 `Status` 的具体值，可能编译出错。

**启用 `isolatedModules` 后**：

```json
{
  "compilerOptions": {
    "isolatedModules": true
  }
}
```

TypeScript 会禁止一些危险的语法：

```typescript
// ❌ const enum（需要内联值）
const enum Color { Red, Green }

// ❌ 仅类型导入的引用（容易混淆）
export type User = { name: string };

// ✅ 改成 enum 或使用 import type
import type { User } from './user';
```

**使用场景**：
- 项目使用 Babel 编译，必须启用
- 使用 Metro（React Native）编译，必须启用
- 使用纯 tsc 编译可选，但启用它让代码更兼容

#### (h) esModuleInterop — ESM/CJS 互操作性

**问题**：TypeScript 默认不会为 CommonJS 导入添加额外逻辑，这在 ESM/CJS 混用时容易出错。

```typescript
// node_modules/lodash/index.js（CommonJS 导出）
module.exports = { map, filter }

// 你的代码（ESM 风格导入）
import _ from 'lodash';  // 期望 _ 是整个 lodash 对象
```

**不启用 esModuleInterop**：TypeScript 认为 `_` 是 `{ map, filter }`，导致 `_.map` 无效。

**启用后**：TypeScript 自动帮你处理这种差异，编译器生成必要的转接代码。

```json
{
  "compilerOptions": {
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true  // 允许 default import
  }
}
```

现代项目几乎都需要启用这两个选项。

## 2、兼容性和使用场景
1、Node.js 环境: Node.js 支持 ES6 的大部分语言特性，并且默认使用 CommonJS 模块系统。因此，target: "ES6" 和 module: "CommonJS" 是一个常见的组合，这有历史原因也有CommonJS在Node.js中的优势。比如 CommonJS 模块系统是同步加载的，这与 Node.js 的单线程、事件驱动架构非常契合。同步加载使得模块在加载时可以立即使用，这对于服务器端应用程序来说非常方便，因为它们通常在启动时加载所有必要的模块。

2、前端环境: 浏览器环境通常使用 ES6 模块系统（import 和 export），但可以通过构建工具（如 Webpack、Babel）将 ES6 模块转换为其他模块系统（如 UMD 或 CommonJS），以实现兼容性。你可能疑惑，既然`tsconfig`已经可以配置编译后的javascript版本和模块系统了，为什么还需要Babel等编译工具呢？其实，对于很多项目如nodejs项目，一般就用只用`tsconfig`。但是Babel、Webpack有更加强大的功能，比如Webpack\Babel可以优化包size，Babel支持最新的 ECMAScript 标准和实验性特性，并且他们还可以通过插件扩展功能等。下面会进行介绍(后续展开介绍)！

最后，关于使用CommonJS或者ES6等不同模块系统的问题，具体见“模块化”文档。

# 二、package.json 的字段配置

当开发一个 npm 包或库时，需要在 `package.json` 中配置特定字段，告诉使用者应该如何引入这个包。这些字段控制了不同的导入方式使用不同的代码。

## 1、main / module / exports 字段

### (1) main 字段（CommonJS 入口）
- `"main": "lib/index.js"` — 当使用 `require()` 导入时，Node.js 会加载这个文件。
- 通常指向编译后的 **CommonJS** 代码。

### (2) module 字段（ESM 入口）
- `"module": "es/index.js"` — 当打包工具（如 Webpack）使用 `import` 导入时，优先加载这个文件。
- 通常指向编译后的 **ESM** 代码。
- 这个字段是非标准的（由 Webpack 等工具定义），但被广泛支持。

### (3) exports 字段（条件导出，推荐）
`exports` 是现代的标准字段，用于定义条件导出。可以根据使用者的导入方式、环境等返回不同的代码。这比 `main` 和 `module` 更灵活、更标准。

```json
{
  "name": "my-library",
  "exports": {
    ".": {
      "import": "./es/index.js",      // import 导入使用 ESM
      "require": "./lib/index.js",    // require 导入使用 CommonJS
      "default": "./lib/index.js"     // 其他情况默认用 CommonJS
    },
    "./utils": {
      "import": "./es/utils.js",
      "require": "./lib/utils.js"
    }
  },
  "main": "lib/index.js",     // 备用
  "module": "es/index.js"     // 备用
}
```

#### exports 的工作原理
```ts
// 用户代码
import { func } from 'my-library';      // 使用 es/index.js
const func = require('my-library');     // 使用 lib/index.js

// 也可以导入子路径
import { util } from 'my-library/utils';  // 使用 es/utils.js
```

#### 关键行为：exports 限制访问范围

一旦定义了 `exports`，只有显式声明的内容能被外部访问。

```typescript
// ❌ 如果没有 exports 中的 .，这样导入会失败
import foo from 'my-lib';

// ✅ 需要按 exports 定义的路径导入
import foo from 'my-lib/react';
```

**理解了这个的价值**：
- 为什么有些库以前能用 `import { x } from 'lib/src/xxx'`，升级后不行了——因为加了 `exports` 限制
- 设计库的公共 API 边界，明确哪些内部模块不该对外暴露
- 知道 `exports` 优先级高于 `main` 和 `module`

#### 使用场景
- **web 应用**：优先使用 ESM（`import`），打包工具可以进行 tree-shaking 优化。
- **Node.js 应用**：根据 Node.js 版本选择 CommonJS 或 ESM。
- **库开发者**：同时发布 CommonJS 和 ESM 两个版本，兼容不同的使用场景。

### (4) types 字段（TypeScript 类型声明）
- `"types": "lib/index.d.ts"` — 指向 TypeScript 类型声明文件。
- 当使用 TypeScript 的项目导入你的库时，会自动加载这个类型文件，获得类型提示。

```json
{
  "name": "my-library",
  "main": "lib/index.js",
  "types": "lib/index.d.ts",
  "exports": {
    ".": {
      "import": "./es/index.js",
      "require": "./lib/index.js",
      "types": "./lib/index.d.ts"
    }
  }
}
```

### (5) sideEffects — 告诉打包器哪些文件有副作用

**概念**：副作用（Side Effects）指代码执行时的额外行为，而不仅仅是导出。

**典型的副作用代码**：

```typescript
// polyfill.js — 有副作用
// 这个模块没有导出，但执行时必须打补丁全局对象
Object.defineProperty(Array.prototype, 'flat', {
  value: function() { /* ... */ }
});

// global.css — 有副作用
// 没有导出，但 CSS 必须被注入到页面
body { margin: 0; }

// 纯函数模块 — 无副作用
export function add(a, b) {
  return a + b;
}
```

**问题场景**：

```typescript
// 使用者导入 polyfill
import './polyfill';  // 这行有副作用，必须执行

// 打包器看到"没有使用任何导出"，可能把整行删掉
```

**`sideEffects` 配置的作用**：

```json
// 完全无副作用
{
  "sideEffects": false  // 打包器可以安全删除所有未使用的模块
}

// 某些文件有副作用
{
  "sideEffects": [
    "*.css",           // 样式文件有副作用
    "*.scss",
    "src/polyfill.js"  // 特定的 polyfill 文件
  ]
}
```

**打包器的行为**：

```typescript
// 如果 sideEffects: false，打包器会删除这些代码
import './unused-module';      // ❌ 整行删除
import { unused } from './x';  // ❌ unused 删除

// 如果 sideEffects 包含该文件，会保留
import './global.css';         // ✅ 保留（因为 *.css 在列表中）
```

**理解了这个的价值**：
- 写库时正确配置 `sideEffects`，让使用者的打包器能正确 tree-shake
- 知道为什么 `import './global.css'` 有时被误删，因为打包器把 CSS 当无副作用模块
- 能排查生产包体积异常增大的问题（可能是 `sideEffects` 配置错了）
- 在库的 package.json 中明确说明哪些入口文件不能被 tree-shake

#### 完整 package.json 示例
```json
{
  "name": "my-ui-library",
  "version": "1.0.0",
  "description": "A React UI component library",
  "main": "lib/index.js",
  "module": "es/index.js",
  "types": "lib/index.d.ts",
  "sideEffects": [
    "*.css",
    "src/setup.js"
  ],
  "exports": {
    ".": {
      "import": "./es/index.js",
      "require": "./lib/index.js",
      "types": "./lib/index.d.ts",
      "default": "./lib/index.js"
    },
    "./button": {
      "import": "./es/components/Button/index.js",
      "require": "./lib/components/Button/index.js",
      "types": "./lib/components/Button/index.d.ts"
    }
  },
  "files": [
    "lib",
    "es",
    "dist"
  ]
}
```

# 三、Webpack/Metro

metro是react-native的默认打包工具，webpack是web的默认打包工具。`Webpack`和`metro`打包工具在打包时会进行静态分析，没有被引用到的模块时不会被打包进最终的输出文件的。
## 1、分包
### (1) require
1. 模块系统：require 是 CommonJS 模块系统的一部分，最初用于 Node.js。
2. 同步加载：require 是同步的，这意味着模块在调用时立即加载并执行。
3. 动态加载：require 可以在代码执行过程中动态加载模块。这意味着您可以根据条件决定是否加载某个模块。
4. Node.js 环境：require 在 Node.js 中广泛使用，但在浏览器环境中不直接支持，但通过使用工具（如 Browserify 或 Webpack），可以将 CommonJS 模块转换为浏览器兼容的代码。

因为 Node.js 运行在服务器端，它可以在运行时访问文件系统并根据需要加载模块。所以，在 Node.js 环境中，如果条件不满足，require 语句不会被执行，模块不会被加载。但是，前端需要打包，所以只能在静态分析阶段判断模块是否需要打包到增量中。通常，require模块都会被打包进bundle中，且不会分包。

### (2) import()
1. 模块系统：import 是 ES6（ES2015）引入的模块系统的一部分，是现代 JavaScript 的标准。
2. 静态分析：import 是静态的，模块在编译时就被解析。这允许工具进行更好的优化，如 tree-shaking。
3. 顶层使用：import 必须在模块的顶层使用，不能在条件语句或函数中动态调用。
4. 广泛支持：现代浏览器和工具链（如 Babel 和 Webpack）都支持 ES6 模块。

还有一点，对于动态加载和代码分割，在现代前端开发中推荐使用 `import()` 语法（ES6语法）。在使用动态 `import()` 语法时，Webpack 会将被动态导入的模块视为代码分割点。这意味着这些模块会被打包成单独的代码块（chunk），这些代码块会在运行时通过网络请求按需加载，而不是包含在主代码包中。这种方式允许按需加载模块，而不是在应用启动时加载所有代码。
```js
import('./module').then(module => {
  // 使用模块
  const myModule = module.default;
});
```

但是，Metro 是 React Native 的打包工具，它与 Webpack 有一些不同的特性。Metro 目前不支持像 Webpack 那样的动态 import() 进行代码分割和按需加载。它将所有 JavaScript 代码打包成一个 bundle 文件，以便在移动设备上高效加载。

**## 1、分包**
-------------------------------------------------------------------------------------------------------
导出方式 × 打包配置	       Webpack (tree-shaking)	      Metro(开InlineRequire)	  Metro(关InlineRequire)
-------------------------------------------------------------------------------------------------------
ES Module 导出            ✅ 按需打包(Tree-shaking生效)    ❌ 全量打包                 ❌ 全量打包
import {A} from '@test'	 ❌ 立即加载                     ✅ 按需加载                 ❌ 立即加载
                         
-------------------------------------------------------------------------------------------------------
Getter 导出方式            ❌ 全量打包(Tree-shaking无效)   ❌ 全量打包                 ❌ 全量打包
                         ✅ 按需加载                     ✅ 按需加载                 ✅ 按需加载
-------------------------------------------------------------------------------------------------------
直接路径导入                ✅ 按需打包                    ✅ 按需打包                 ✅ 按需打包
import A from 
'@test/src/components/A'  ❌ 立即加载                    ✅ 按需加载                 ❌ 立即加载
-------------------------------------------------------------------------------------------------------

值得注意的是，针对ES Module 导出，Metro开启InlineRequire，并非完全没有tree-shaking功能。对于简单导出但未使用的模块，静态分析阶段，可能会被移除。如下：
```ts
// 这种情况 Metro 可能只打包 Page3
import Page3 from './Page3';
import Page4 from './Page4'; // 未使用，可能被移除

const config = {
  page: Page3 // 只使用了 Page3
};
```

这里对metro的静态分析能力做一个分析：
### 1、情况1：直接导入具体路径
```ts
import PriceInfo from '@ctrip/xtaro-hotel-components-crn/src/components/price-info'
```
Metro 的分析过程：
1. 解析 import 语句，找到目标文件 ./src/components/price-info
2. 分析该文件的依赖关系
3. 只打包 price-info 组件及其直接依赖

### 2、通过动态 getter 导入
```ts
import PriceInfo from '@ctrip/xtaro-hotel-components-crn'
```
```ts
// 组件index.ts
const CRN = {
  get PriceInfo() {
    const module = require('./src/components/price-info');
    return module.default;
  },
  get SegmentControl() {
    const module = require('./src/components/segment-control');
    return module.default;
  }
}
```
Metro 也无法在静态阶段确定我们不会使用 CRN.SegmentControl。因为 CRN 对象被导出，其他模块可能以“动态方式”使用它。因此，Metro 会保守地将所有可能用到的模块都打包。比如下面代码就用到了SegmentControl，但是静态分析分析不出来：
```ts
// 或者通过循环访问所有组件
Object.keys(CRN).forEach(key => {
    const Component = CRN[key]; // 可能访问所有getter
    // 做一些操作...
});
```
Metro 的按需打包依赖于静态分析，而 getter 函数和动态 require 破坏了静态可分析性。为了获得最佳的 tree-shaking 效果，应该使用直接的 ES6 模块导出语法。

## 2、Webpack 配置示例

Webpack 是前端最流行的打包工具。基础配置包括入口、出口、加载器和插件。

```js
// webpack.config.js
const path = require('path');

module.exports = {
  mode: 'production',  // 'production' 或 'development'
  
  // 入口文件
  entry: './src/index.js',
  
  // 输出配置
  output: {
    filename: 'bundle.[contenthash].js',  // [contenthash] 会根据内容生成哈希
    path: path.resolve(__dirname, 'dist'),
    clean: true,  // 清理前面的输出
  },
  
  // 加载器（Loaders）- 处理不同类型的文件
  module: {
    rules: [
      {
        test: /\.jsx?$/,  // 匹配 .js 和 .jsx 文件
        exclude: /node_modules/,
        use: {
          loader: 'babel-loader',  // 使用 Babel 转换 JavaScript
          options: {
            presets: ['@babel/preset-env', '@babel/preset-react']
          }
        }
      },
      {
        test: /\.tsx?$/,  // 匹配 TypeScript 文件
        use: 'ts-loader',
        exclude: /node_modules/
      },
      {
        test: /\.css$/,
        use: ['style-loader', 'css-loader']  // 加载器链从右到左执行
      },
      {
        test: /\.(png|jpg|gif)$/,
        type: 'asset',  // Webpack 5 内置资源处理
        parser: {
          dataUrlCondition: {
            maxSize: 8 * 1024  // 小于 8KB 的图片转为 base64
          }
        }
      }
    ]
  },
  
  // 插件（Plugins）- 执行特定的任务
  plugins: [
    new (require('html-webpack-plugin'))({
      template: './src/index.html',  // 以此模板生成 HTML
      minify: {
        removeComments: true,  // 移除注释
        collapseWhitespace: true  // 移除空白
      }
    })
  ],
  
  // 优化配置
  optimization: {
    splitChunks: {
      cacheGroups: {
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendors',
          priority: 10,  // 优先级，值越大优先级越高
          chunks: 'all'  // 对所有类型的 chunk 进行分割
        }
      }
    }
  },
  
  // 开发服务器配置
  devServer: {
    port: 3000,
    open: true,  // 自动打开浏览器
    hot: true,   // 热更新
    historyApiFallback: true  // 处理 React Router 等前端路由
  },
  
  // 开发工具 - 生成 source map 用于调试
  devtool: process.env.NODE_ENV === 'production' ? false : 'source-map',
  
  // 模块解析配置
  resolve: {
    extensions: ['.js', '.jsx', '.ts', '.tsx'],  // 导入时可以省略扩展名
    alias: {
      '@': path.resolve(__dirname, 'src'),  // 路径别名
      '@components': path.resolve(__dirname, 'src/components')
    }
  }
};
```

#### 关键概念解释

**Loaders（加载器）**
- 用于处理非 JavaScript 文件。Webpack 本身只能处理 JavaScript，loaders 将其他格式的文件转换为 JavaScript 模块。
- 常见 loaders：`babel-loader`（转换 JS）、`ts-loader`（转换 TypeScript）、`css-loader`（处理 CSS）、`file-loader`（处理图片）。

**Plugins（插件）**
- 用于执行更广泛的任务，贯穿整个构建过程。
- 常见插件：`HtmlWebpackPlugin`（生成 HTML）、`MiniCssExtractPlugin`（提取 CSS）、`TerserPlugin`（代码压缩）。

**Code Splitting（代码分割）**
- 将代码拆分成多个 chunk，按需加载。常见策略：
  - 分离 vendor（第三方库）
  - 分离 runtime（Webpack 运行时）
  - 分离 common（多个 entry 共用代码）

## 3、Metro 配置示例

Metro 是 React Native 的打包工具。配置通常在 `metro.config.js` 中。

```js
// metro.config.js
const {getDefaultConfig} = require('metro-config');

module.exports = (async () => {
  const {
    resolver: {sourceExts, assetExts}
  } = await getDefaultConfig();
  
  return {
    // 项目根目录
    projectRoot: __dirname,
    
    // 工作区配置（monorepo）
    watchFolders: [
      `${__dirname}/node_modules`,
      `${__dirname}/packages`

    // 入口点

    // 输出路径

      // 输出格式
      getModuleId: (modulePath) => {
        // 自定义模块 ID 生成
        return modulePath;
      }
    },
    
    // 转换器配置
    transformer: {
      // 是否启用 InlineRequire
      inlineRequires: {
        blacklist: [
          // 某些模块不应该被 inline
        ]
      },
      // Babel 配置
      babelTransformerPath: require.resolve('metro-react-native-babel-transformer'),
      getTransformOptions: async () => ({
        transform: {
          experimentalImportSupport: false,
          inlineRequires: true  // 启用 InlineRequire
        }

    },
    
    // 解析器配置
    resolver: {

        '@': `${__dirname}/src`,
        '@components': `${__dirname}/src/components`
      }
    },
    
    // 最大工作进程数
    maxWorkers: 4
  };

```

#### Metro vs Webpack 的关键差异
- **输出格式**：Webpack 生成浏览器可用的代码，Metro 生成 React Native 可用的代码。
- **代码分割**：Webpack 支持动态 `import()` 分割，Metro 不支持（生成单一 bundle）。
- **性能**：Metro 针对移动开发优化，启动更快。

## 4、lib / es / dist

（1）`dist` 是Webpack/metro的产物；`es`和`lib`是`react`或者`react native`项目中两个常见的文件夹，用于存放 JavaScript 代码;   

（2）`es`文件夹中存放的是经过Babel转译后的`ESModules代码`;   
  
（3）`lib`文件夹中存放的是经过Babel转译后的`CommonJS代码`;  

通常，在`React Native`项目中，开发者可以选择使用 `es` 文件夹中的代码进行开发和调试，而将 `lib` 文件夹中的代码用于构建和发布应用程序。这样可以在开发过程中享受到更好的开发体验，同时在生产环境中获得更好的兼容性。

## 3、引入es还是lib？

如果我们使用一个依赖，且该依赖目录包含`es`和`lib`两个文件。那使用`import` 引入就是使用 `es`文件夹下内容，`require` 引入就是使用 `lib`文件夹下内容。   

可以试验一下，我们引入一个`npm依赖`，如果使用 vscode 点击会发现跳转到了`lib`文件下，是因为 vscode 默认是 node 环境，链接时按照 `require` 来链接；但最终打包 Webpack 是依据你当前导入语句来决定的。目前，开发基本都是使用 ES6，即使用 import 的方式导入模块，并使用 Babel 进行向下兼容。在 Node.js 中，模块是通过 CommonJS 规范来定义和导出的。虽然从 Node.js 12 版本开始，Node.js 已经开始支持 ES Modules 了。但是，Node.js 仍然默认使用 CommonJS 规范来加载模块。如果你想在 Node.js 中使用 ES Modules，需要在文件中添加 `.mjs` 扩展名，并使用 `--experimental-modules` 标志来启用实验性的 ES Modules 功能。

# 七、tsc vs Babel：职责分离

在前面介绍 Babel 和 `isolatedModules` 时，有个隐藏的问题：**TypeScript 编译器做了两件事**。

## 1、两个不同的任务

TypeScript 编译器（tsc）同时做：
1. **类型检查** — 检测 ts 文件的类型错误
2. **代码转换** — 把 ts 代码转成 js 代码

```typescript
// 源代码
function add(a: number, b: number): number {
  return a + b;
}

add("1", "2");  // ❌ tsc 检测到类型错误
```

但这两个操作的频率和需求不同：

| 操作 | 频率 | 性能需求 | 谁来做 |
|------|------|----------|--------|
| **类型检查** | 每次保存都做 | 可以容忍 1-2 秒 | tsc / TypeScript 服务 |
| **代码转换** | 每次保存都做 | 必须快（毫秒级） | Babel |

## 2、为什么要分离？

### 问题场景

```bash
# 传统做法：用 tsc 做一切
tsc src --outDir dist

# 这个过程：
# 1. 解析 TS 文件，建立完整类型系统（慢！）
# 2. 逐个转换代码（中等）
# 总耗时：2-5 秒
```

在大型项目中，一次改动要等 5 秒才能测试，效率很低。

### 分离后的架构

```
开发时：
  输入：TS 代码
    ↓
  [类型检查线程] — tsc in watch mode（可以保证准确）
  [代码转换线程] — Babel（尽快输出 JS）
    ↓
  输出：JS 代码（立即）+ 类型错误（稍后）
```

**好处**：
- 代码转换不需要等待完整的类型检查（Babel 不做类型检查）
- 大型项目的编辑-刷新循环从 5 秒降到 500ms
- 类型检查在后台进行，不阻塞开发

### 实际配置示例

**package.json**：
```json
{
  "scripts": {
    "dev": "concurrently \"tsc --watch --noEmit\" \"babel src --watch --out-dir dist\"",
    "build": "tsc && babel src --out-dir dist",
    "check-types": "tsc --noEmit"
  }
}
```

**babel.config.js**：
```js
module.exports = {
  // Babel 不做类型检查，只做转换
  presets: [
    ["@babel/preset-env", { useBuiltIns: "usage" }],
    "@babel/preset-typescript"  // 直接删掉 TS 语法，不检查类型
  ]
}
```

关键点：`@babel/preset-typescript` 只是删掉类型注解，不进行类型检查。

## 3、CI/CD 中的应用

在持续集成中，通常需要：

```bash
# 并行执行
npm run check-types &      # tsc --noEmit（类型检查）
npm run build &            # 编译输出
npm run test &             # 运行测试
wait

# 或串行执行，优化速度
npm run lint                # 快速检查（eslint）
npm run build               # Babel 转换
npm run check-types         # 类型检查（可选，等build完后）
npm run test                # 测试
```

关键认识：不是所有编译步骤都必须串行，类型检查可以和代码编译并行。

打包优化的目标是减小包体积、提升加载速度。常见的优化策略包括：

## 1、Tree-shaking

**概念**：自动检测和移除未使用的代码。只有当代码是**静态可分析**的时，tree-shaking 才能工作。

### 如何启用 tree-shaking

#### a) 使用 ES6 模块语法
```ts
// ✅ 支持 tree-shaking（静态导出）
export const unused = () => {};
export const used = () => {};

import { used } from './module';  // unused 会被移除
```

```ts
// ❌ 不支持 tree-shaking（动态导出）
const obj = {
  unused: () => {},
  used: () => {}
};
export default obj;  // 无法判断哪个属性被使用
```

#### b) Webpack 配置
```js
// webpack.config.js
module.exports = {
  mode: 'production',  // 必须是 production 模式
  entry: './src/index.js',
  output: {
    filename: 'bundle.js',
    path: path.resolve(__dirname, 'dist')
  },
  optimization: {
    usedExports: true,  // 标记未使用的导出
    sideEffects: false  // 假设没有副作用，允许完全移除模块
  }
};
```

#### c) package.json 配置
```json
{
  "sideEffects": false  // 告诉 Webpack 此包没有副作用
}
```

**副作用（Side Effects）**：指代码执行时的额外行为，例如修改全局变量、导入样式文件等。如果模块有副作用，不能被完全移除，即使它没有被导入。

### Tree-shaking 的限制
- 仅支持 ES6 模块（import/export），CommonJS 的 require/module.exports 不支持。
- 动态导入（如 `require(variable)`）无法 tree-shake。
- 对象的 getter 方法、高阶函数等破坏静态可分析性，会导致 tree-shaking 失效。

## 2、代码分割（Code Splitting）

**概念**：将大的代码包分割成多个小块，按需加载，减少初始加载时间。

### Webpack 配置
```js
module.exports = {
  entry: './src/index.js',
  output: {
    filename: '[name].bundle.js',  // [name] 用实际名称替换
    path: path.resolve(__dirname, 'dist')
  },
  optimization: {
    splitChunks: {
      chunks: 'all',  // 对所有 chunk 进行分割
      cacheGroups: {
        // vendor chunk - 第三方库
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendors',
          priority: 10
        },
        // common chunk - 多个 entry 共用代码
        common: {
          minChunks: 2,  // 至少被 2 个 chunk 使用
          priority: 5,
          reuseExistingChunk: true
        }
      }
    }
  }
};
```

### React 动态导入
```ts
// 使用 React.lazy 进行代码分割
import React, { Suspense, lazy } from 'react';

const HeavyComponent = lazy(() => import('./HeavyComponent'));

export default function App() {
  return (
    <Suspense fallback={<div>Loading...</div>}>
      <HeavyComponent />
    </Suspense>
  );
}
```

## 3、压缩和混淆

Webpack 在 production 模式下会自动启用 Terser 插件进行代码压缩。

```js
module.exports = {
  mode: 'production',  // 自动启用压缩
  optimization: {
    minimize: true,
    minimizer: [
      new (require('terser-webpack-plugin'))({
        terserOptions: {
          compress: {
            drop_console: true  // 移除 console 语句
          },
          mangle: true  // 混淆变量名
        }
      })
    ]
  }
};
```

## 4、包体积分析

### webpack-bundle-analyzer
用于可视化分析打包结果，找出体积最大的模块。

```js
// webpack.config.js
const BundleAnalyzerPlugin = require('webpack-bundle-analyzer').BundleAnalyzerPlugin;

module.exports = {
  // ... 其他配置
  plugins: [
    new BundleAnalyzerPlugin({
      analyzerMode: 'static',  // 生成静态 HTML 报告
      openAnalyzer: true
    })
  ]
};
```

打包后，会自动打开 HTML 报告，展示每个模块的大小。

### 常见优化建议
- **移除未使用的依赖** — 定期清理 package.json。
- **使用 CDN** — 将大型第三方库（如 React）上传到 CDN，通过 externals 配置排除打包。
- **懒加载** — 按路由或用户交互进行代码分割和加载。
- **缓存** — 使用 contenthash 在文件名中，只有内容改变时才重新下载。

# 三、Babel
## 1、babel的转换
首先，ES6的升级地方有很多，比如箭头函数、类、模板字符串、解构、等，参数默认值、let/const、for…of、map/set等（详情见https://www.babeljs.cn/docs/learn）。babel会对一些新句法做转换，比如箭头函数、类、let/const等。需要注意的是，Babel 默认只转换新的 JavaScript 句法，而不转换新的 API，比如`Iterator`、`Generator`、`Set`、`Map`、`Proxy`、`Reflect`、`Symbol`、`Promise`等全局对象，以及一些定义在全局对象上的方法（比如`Object.assign`）都不会转码。不过，Babel提供了polyfill来转换这些类型。

举例来说，ES6 在`Array`对象上新增了`Array.from`方法。Babel 就不会转码这个方法。如果想让这个方法运行，必须使用`babel-polyfill`，为当前环境提供一个垫片。否则比如代码：   

`colors = Array.from(str.matchAll(regex), match => match[1]);`   

在commonJS环境下会报错。(polyfill 配置方式见[bable配置polyfill](https://www.babeljs.cn/docs/babel-preset-env))。但是，polyfill会增加额外的空间和性能开销！

## 2、ES6新增API
ES6新增的API有Map、Set、Promise等，新增的API方法有很多，分类如下：
### 1、数组（Array）新增 API
Array.from()	将类数组对象或可迭代对象转为数组。
Array.of()	创建一个包含任意数量元素的新数组。
arr.find(callback)	返回第一个符合条件的元素。
arr.findIndex(callback)	返回第一个符合条件的元素的索引。
arr.fill(value)	用固定值填充数组。
arr.copyWithin()	复制数组的一部分到同一数组的另一位置。
arr.entries()	返回数组的键值对迭代器。
arr.keys()	返回数组的键（索引）迭代器。
arr.values()	返回数组的值迭代器。

### 2、对象（Object）新增 API
Object.assign(target, ...sources)	合并多个对象的属性到目标对象（浅拷贝）。
Object.is(value1, value2)	更严格的相等性比较（替代 ===）。
Object.setPrototypeOf(obj, prototype)	设置对象的原型。
Object.getPrototypeOf(obj)	获取对象的原型。
Object.keys() / Object.values() / Object.entries()	返回对象自身可枚举属性的键、值、键值对。

### 3、函数（Function）新增特性
箭头函数	() => {}，简化函数语法，绑定 this。
参数默认值	function(a = 1) {}
剩余参数	function(...args) {}
扩展运算符	Math.max(...array)

### 4、字符串（String）新增 API
str.startsWith(search)	检查字符串是否以某个子串开头。
str.endsWith(search)	检查字符串是否以某个子串结尾。
str.includes(search)	检查字符串是否包含某个子串。
str.repeat(n)	重复字符串 n 次。
模板字符串	`Hello ${name}`，支持多行和插值。

### 5、数字（Number）新增 API
Number.isNaN()	更安全的 NaN 检测。
Number.isFinite()	检查是否为有限数。
Number.isInteger()	检查是否为整数。
Number.EPSILON	表示最小精度（用于浮点数比较）。
Math.trunc()	去除小数部分（直接截断）。
Math.sign()	返回数值的符号（正、负、零）。

### 6、其他新增 API
Symbol	Symbol()	创建唯一的值，用作对象属性的键。
迭代器（Iterator）	[Symbol.iterator]	定义对象的迭代行为（如 for...of）。
生成器（Generator）	function* 和 yield	生成迭代器的函数。
模块化（Module）	import / export	模块导入导出语法。


## 3、babel常见依赖
### (1) 基础依赖
1. @babel/core： Babel 的核心引擎，负责代码解析、转换和生成。所有其他 Babel 工具都依赖于它。

### (2) 预设（Presets）
预设（Preset）是一个 Babel 插件的集合。Babel 本身只是一个代码转换引擎，具体如何转换由插件决定。预设将多个常用插件打包在一起，无需手动逐个添加。常见预设有：

1. **@babel/preset-env**：自动根据目标环境转译 ES6+ 代码。这是最常用的预设。
   - 它会自动检测你需要支持的浏览器或环境，然后只转换该环境不支持的 ES6+ 特性。
   - 例如，如果目标是 ES2020，那么 `const` 和 `let` 就不需要转换（因为 ES2020 已支持），但 `?.`（可选链操作符）可能需要特殊处理。

2. **@babel/preset-react**：支持 React 的 JSX 语法转换，将 JSX 转换为 `React.createElement` 调用。

3. **@babel/preset-typescript**：支持 TypeScript 语法。让 Babel 能够解析和转换 TypeScript 代码（注意：只做转换，不做类型检查）。

4. **@babel/preset-flow**：支持 Flow 类型检查语法。

预设（Presets）需要在 Babel 的配置文件中声明（如 babel.config.json、.babelrc 或其他支持的配置文件格式）。

#### babel.config.json 示例
```json
{
  "presets": [
    ["@babel/preset-env", { 
      "targets": "> 0.5%, last 2 versions",
      "useBuiltIns": "usage",
      "corejs": 3
    }],
    "@babel/preset-react",
    "@babel/preset-typescript"
  ]
}
```

### (3) 插件（Plugins）
如果预设不包含你需要的转换，可以单独添加插件。插件是针对特定语法特性的转换器。常见插件有：

1. **@babel/plugin-transform-runtime**：最常用的插件。
   - **用途**：避免编译后代码中重复的辅助代码，减小打包体积。
   - **原理**：当 Babel 转换某些语法时，会生成一些辅助函数（如 `__extends`、`__awaiter` 等）。如果多个文件都有类似的转换，这些辅助函数会重复出现，浪费空间。该插件会把这些辅助函数提取到 `@babel/runtime` 包中，所有文件都共用这个包中的辅助函数。
   - 例如，在没有该插件时：
     ```ts
     // 源代码
     class Animal {}
     class Dog extends Animal {}
     
     class Cat extends Animal {}
     ```
     编译后会生成两份 `__extends` 辅助函数。使用插件后，两个文件都会引用同一个 `@babel/runtime` 中的 `__extends`。

2. **@babel/plugin-proposal-decorators**：支持装饰器（Decorators）语法。
   - 装饰器是一种 TypeScript/JavaScript 的高级特性，用于修饰类或方法。
   ```ts
   @logged
   class MyClass {
     @validate
     myMethod(param: string) {}
   }
   ```

3. **@babel/plugin-proposal-class-properties**：支持类属性的直接赋值。
   ```ts
   class MyClass {
     // 不需要在 constructor 中赋值
     count = 0;
     
     increment = () => {
       this.count++;
     };
   }
   ```

4. **@babel/plugin-proposal-nullish-coalescing-operator**：支持空值合并操作符 `??`。
   ```ts
   const value = nullableValue ?? defaultValue;
   ```

5. **@babel/plugin-proposal-optional-chaining**：支持可选链操作符 `?.`。
   ```ts
   const name = user?.profile?.name;
   ```

#### 配置示例
```json
{
  "presets": [
    ["@babel/preset-env", { 
      "useBuiltIns": "usage",
      "corejs": 3
    }],
    "@babel/preset-react",
    "@babel/preset-typescript"
  ],
  "plugins": [
    ["@babel/plugin-transform-runtime", {
      "corejs": 3
    }],
    ["@babel/plugin-proposal-decorators", { "legacy": true }],
    "@babel/plugin-proposal-class-properties",
    "@babel/plugin-proposal-nullish-coalescing-operator",
    "@babel/plugin-proposal-optional-chaining"
  ]
}
```

### (4) useBuiltIns 详解 — Polyfill 的现代方案

在介绍 polyfill 时，提到了三种模式。这里详细解释：

#### a) useBuiltIns: false（默认）
不自动注入 polyfill。如果代码中使用了新 API（如 `Array.from`），在不支持的环境中会报错。

```ts
// 源代码
const arr = Array.from([1, 2, 3]);
const set = new Set([1, 2, 3]);

// 编译后（useBuiltIns: false）
var arr = Array.from([1, 2, 3]);  // Array.from 在旧浏览器中不存在，会报错
var set = new Set([1, 2, 3]);     // Set 在旧浏览器中不存在，会报错
```

#### b) useBuiltIns: "entry"
需要在代码入口手动导入 `core-js/stable` 和 `regenerator-runtime`，Babel 会根据目标环境替换为具体的 polyfill。

```ts
// babel.config.json
{
  "presets": [
    ["@babel/preset-env", {
      "useBuiltIns": "entry",
      "corejs": 3
    }]
  ]
}

// 应用入口文件（src/index.js）
import 'core-js/stable';
import 'regenerator-runtime/runtime';

// 你的应用代码
const arr = Array.from([1, 2, 3]);
```

编译后，Babel 会替换成：
```js
import 'core-js/es/array/from';
import 'core-js/es/set';
import 'regenerator-runtime/runtime';

const arr = Array.from([1, 2, 3]);
```

**优点**：清晰，一次性导入所有 polyfill。
**缺点**：可能引入不需要的 polyfill，增加包体积。

#### c) useBuiltIns: "usage"（推荐）
Babel 会自动检测代码中使用的新 API，并按需注入相应的 polyfill。不需要手动导入。

```ts
// babel.config.json
{
  "presets": [
    ["@babel/preset-env", {
      "useBuiltIns": "usage",
      "corejs": 3
    }]
  ]
}

// 源代码（无需手动导入 polyfill）
const arr = Array.from([1, 2, 3]);
const set = new Set([1, 2, 3]);
```

编译后：
```js
import 'core-js/es/array/from';
import 'core-js/es/set';

const arr = Array.from([1, 2, 3]);
const set = new Set([1, 2, 3]);
```

**优点**：只引入实际用到的 polyfill，包体积最小。
**缺点**：需要 Babel 正确识别，对于动态代码可能识别不了。

#### 如何选择？
- 新项目优先使用 `"usage"` 模式，按需加载 polyfill。
- 如果 `"usage"` 模式识别有问题，改用 `"entry"` 模式。
- `false` 模式适合现代浏览器项目，不需要 polyfill。

### (5) @babel/preset-env 的 targets 配置

`targets` 决定了 Babel 的转换粒度。指定得越具体，转换得越少，包体积越小。

```json
{
  "presets": [
    ["@babel/preset-env", {
      "targets": {
        "browsers": [
          "> 0.5%",           // 全球浏览器市场占有率 > 0.5%
          "last 2 versions",  // 每个浏览器的最后 2 个版本
          "not dead"          // 排除已停止维护的浏览器
        ],
        "node": "14.0"        // 或指定 Node.js 版本
      }
    }]
  ]
}
```

也可以简写：
```json
{
  "presets": [
    ["@babel/preset-env", {
      "targets": "> 0.5%, last 2 versions"
    }]
  ]
}
```
