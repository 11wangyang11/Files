# 编译打包深度认识指南

## 核心问题

当你写 TypeScript 代码时，它最终需要运行在某个环境里（浏览器、Node.js 等）。但浏览器和 Node.js 环境：
- 支持的 JavaScript 语法不同
- 模块系统不同
- 性能要求不同

**编译打包就是解决这个问题的过程。**

---

## 整体流程

```
┌─ 源代码（TypeScript）
│
├─【第一阶段：编译】
│  ├─ tsc：类型检查 + 转换成 JavaScript
│  ├─ Babel：进一步转换（ES6 → ES5、删除 TS 语法等）
│  └─ 输出：多个版本的 JavaScript（ESM、CJS 等）
│
├─【第二阶段：打包】
│  ├─ 找到所有依赖
│  ├─ 代码优化（tree-shake、压缩等）
│  └─ 生成最终的 bundle
│
└─ 最终产物（可以直接运行的代码）
```

---

## 第一部分：编译阶段

### 为什么需要编译

```
浏览器的限制：
  ✗ 不支持 TypeScript 类型
  ✗ 不支持最新的 ES2025 语法
  ✗ 不支持 CommonJS (require)
  ✗ 不支持导入 .ts 文件

解决方案：编译
  ✓ 删除 TypeScript 类型
  ✓ 转换新语法为旧语法
  ✓ 统一模块系统
  ✓ 处理文件扩展名
```

### 编译涉及的两个工具

#### 1. TypeScript 编译器（tsc）

**职责**：
- 类型检查
- 转换 TypeScript → JavaScript
- 决定**编译后代码的格式**

**关键配置**：

```json
{
  "compilerOptions": {
    "target": "ES2020",      // ← 编译后的 JS 版本
    "module": "ES2020",      // ← 编译后使用什么模块系统
    "jsx": "react-jsx",      // ← 如何处理 JSX
    "declaration": true      // ← 生成 .d.ts 类型文件
  }
}
```

**理解 target 和 module**：

- **target**：决定编译后代码中**函数和语法**的版本
  - target: ES5 → 箭头函数变成普通函数，const 变成 var
  - target: ES2020 → 保留箭头函数和 const（因为 ES2020 支持）

- **module**：决定编译后代码中**导入导出**的写法
  - module: CommonJS → `const X = require('y'); module.exports = ...`
  - module: ES2020 → `import X from 'y'; export default ...`

**重要细节**：tsc 只负责编译，不负责把新 API（如 `Array.from`）转换成旧版本能用的形式。这需要 Babel。

#### 2. Babel（可选）

**职责**：
- 进一步转换 JavaScript（新语法 → 旧语法）
- 添加 polyfill（新 API 的兼容代码）
- 删除仍然存在的 TypeScript 代码

**为什么需要 Babel**：

tsc 只转换语法，不转换 API。例如：

```typescript
// 源代码
const arr = Array.from([1, 2, 3]);

// tsc 编译后（target: ES5）
var arr = Array.from([1, 2, 3]);  // ← 语法转了，但 Array.from 在 ES5 里不存在！

// 需要 Babel + polyfill
var arr = __POLYFILL_Array_from([1, 2, 3]);  // ← 现在能用了
```

**Babel 的关键配置**：

```json
{
  "presets": [
    ["@babel/preset-env", {
      "targets": "> 1%",        // ← 目标环境（浏览器版本）
      "useBuiltIns": "usage"    // ← 按需加 polyfill（不会把所有 polyfill 都加入）
    }]
  ]
}
```

## 第二部分：模块系统与版本管理

### 问题：为什么库要发布多个版本

库发布给不同的使用场景，需要不同的格式：

```
【Node.js 项目】
  └─ 使用 require()
  └─ 需要 CommonJS 版本（lib/）

【浏览器 + 打包工具（Webpack）项目】
  └─ 使用 import
  └─ 需要 ESM 版本（es/）
  └─ 打包工具能进行 tree-shaking（优化）
```

### ESM vs CommonJS

**关键区别**：

```typescript
// ESM（浏览器友好）
export const helper = () => {}
import { helper } from './module'

// 特点：
// ✓ 编译时就能分析依赖（打包工具可以 tree-shake）
// ✓ 浏览器原生支持

// CommonJS（Node.js 标准）
module.exports = { helper }
const { helper } = require('./module')

// 特点：
// ✓ Node.js 原生支持
// ✗ 运行时才分析依赖（打包工具无法 tree-shake）
```

**打包结果的差异**：

```
导入 lodash

ESM 版本：
  打包工具能识别你只用了 map 函数
  最终包大小：2KB

CommonJS 版本：
  打包工具看不懂，整个 lodash 都打进去
  最终包大小：70KB

差异：35 倍！
```

### package.json 如何控制版本

```json
{
  "main": "lib/index.js",           // ← require() 时用这个（CJS）
  "module": "es/index.js",          // ← import 时优先用这个（ESM）
  "exports": {
    ".": {
      "import": "./es/index.js",    // ← import 导入时返回 ESM
      "require": "./lib/index.js"   // ← require 导入时返回 CJS
    }
  }
}
```

**关键认识**：
- `main` 和 `module` 是给打包工具的**建议**
- `exports` 是**强制规则**（优先级最高）
- 当库有 `exports` 时，只有定义过的路径才能导入
- 这保护库的内部实现不被依赖

## 第三部分：打包阶段

### 打包工具的职责

```
【输入】多个 JS 文件 + 依赖关系

【处理过程】
├─ 1. 模块解析：找到所有导入导出的关系
├─ 2. Tree-shaking：删除未使用的代码
├─ 3. 代码分割：决定如何分割成多个 bundle
├─ 4. 优化：压缩、混淆
└─ 5. 输出：生成最终的 bundle

【输出】可以直接在浏览器/Node.js 运行的代码
```

### Tree-shaking 的原理

**前提条件**：
- 代码必须是 ESM（`import`/`export`）
- 打包工具能静态分析（知道哪些代码会被用到）

```typescript
// module.ts
export const used = () => console.log('used')
export const unused = () => console.log('unused')

// app.ts
import { used } from './module'
used()

// 打包结果
// ✓ used 函数被打进去
// ✗ unused 函数被删除（tree-shake）
```

**关键配置**：

在 package.json 里标记哪些文件有"副作用"：

```json
{
  "sideEffects": false        // ← 这个包没有副作用，可以安全删除
}

// 或者

{
  "sideEffects": ["*.css"]    // ← 除了 CSS 文件，其他都没副作用
}
```

**副作用是什么**：

```typescript
// 有副作用的代码（不能删）
import './global-setup'  // 这行虽然不导入东西，但执行时修改了全局状态
Object.defineProperty(Array.prototype, 'flat', {...})

// 无副作用的代码（可以删）
export const helper = () => {}  // 只导出，没有其他影响
```

## 第四部分：关键配置的理由

### 为什么需要 moduleResolution

**问题**：当你写 `import X from 'react'` 时，TypeScript 需要找到这个模块。但怎么找？

```
选项 1：node 规则（传统）
  ├─ 查看 package.json 的 main 字段
  ├─ 加载 lib/index.js（CJS 版本）
  └─ 不支持 tree-shaking

选项 2：bundler 规则（现代）
  ├─ 查看 package.json 的 exports 字段
  ├─ 加载 es/index.js（ESM 版本）
  └─ 支持 tree-shaking
```

**前端项目应该用 bundler**：

```json
{
  "compilerOptions": {
    "moduleResolution": "bundler"
  }
}
```

因为你的代码最终会被 Webpack 等打包工具处理，需要 ESM 才能 tree-shake。

### 为什么需要 isolatedModules

**问题**：TypeScript 编译器能跨文件分析，但 Babel 只能单文件编译。

```
跨文件编译（tsc）：
  看 file1.ts：export const enum Status = { Pending = 0 }
  看 file2.ts：import { Status } from './file1'
  知道 Status.Pending = 0
  ✓ 能正确内联

单文件编译（Babel）：
  编译 file1.ts：const enum 被删除
  编译 file2.ts：看不到 file1.ts 的信息
  不知道 Status.Pending 是多少
  ✗ 编译失败
```

**解决方案**：启用 `isolatedModules`，让 TypeScript 检查代码是否能被单文件编译

```json
{
  "compilerOptions": {
    "isolatedModules": true  // ← 禁止 const enum、某些类型导出等
  }
}
```

这样 IDE 会立即报错，强制你用单文件兼容的写法。

## 第五部分：工作流程

### 开发时

```
你改代码
  ↓
【后台并行】
├─ tsc --watch：检查类型错误（持续进行）
├─ Babel watch：转换代码（快速输出）
└─ 打包工具 watch：生成 bundle（持续进行）
  ↓
开发服务器更新
  ↓
浏览器热刷新
```

**重点**：tsc 只做类型检查，Babel 做真正的代码转换。分开能提高速度。

### 构建时

```
npm run build
  ↓
【顺序执行】
├─ 1. tsc 编译：生成 JS 文件
├─ 2. Babel 转换（可选）：进一步转换
└─ 3. 打包工具：最后的优化和打包
  ↓
dist/ 文件夹（最终产物）
```

---

## 总结：为什么每个配置都必要

| 配置 | 作用 | 为什么必要 |
|------|------|----------|
| target | 编译后的 JS 版本 | 影响代码中函数和语法的写法 |
| module | 编译后的模块系统 | 影响 import/export 的写法 |
| jsx | JSX 转换方式 | React 项目需要 |
| moduleResolution | 怎样找依赖 | 前端用 bundler 才能 tree-shake |
| declaration | 生成 .d.ts | 库发布时用户才有类型提示 |
| isolatedModules | 单文件编译检查 | 用 Babel 时必须，保证兼容性 |

## 实际例子：一个完整的库发布流程

```
源代码（src/）
  ↓
【构建】
├─ tsc：编译成 lib/（CJS） 和 es/（ESM）
├─ tsc：生成 .d.ts 类型文件
└─ Babel：进一步转换（如果需要支持更旧的浏览器）
  ↓
dist/ 结构
├─ lib/          （CommonJS 版本）
│  ├─ index.js
│  └─ index.d.ts
│
└─ es/           （ESM 版本）
   ├─ index.js
   └─ index.d.ts
  ↓
【发布】
package.json 中：
{
  "main": "lib/index.js",
  "module": "es/index.js",
  "types": "lib/index.d.ts",
  "exports": {
    ".": {
      "import": "./es/index.js",
      "require": "./lib/index.js",
      "types": "./lib/index.d.ts"
    }
  }
}
  ↓
npm publish
  ↓
【使用者】
├─ Node.js：require() → 使用 lib/（CJS）
├─ Webpack：import → 使用 es/（ESM）→ tree-shake
└─ TypeScript：获得 .d.ts 类型提示
```
