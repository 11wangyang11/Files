### SPA 适合什么场景（含 MPA 对照）
> 关键认知：**SPA 不是框架属性，是【运行时行为】**。框架（React / Next.js）只是【提供 SPA 能力】，你的代码用 `<Link>` 还是 `location.href` 才决定一次跳转走 SPA 还是 MPA。

## 一、SPA 适合的场景（三个核心特征）
满足以下任意一个，SPA 收益就明显：

```bash
特征                                     说明
─────────────────────────────────────────────────────────────────
① 用户高频操作 / 跨页面状态共享          频繁切 tab、共享购物车、登录态
② 单团队维护 / 紧密关联的页面流程        下单流程：选房 → 资料 → 确认 → 支付
③ 应用化体验需求 (像原生 App)            后台管理系统、协作编辑、实时通讯
```

*典型 SPA 项目*：
```bash
中后台 / Dashboard         ⭐⭐⭐⭐⭐  操作密集、状态多、不需要 SEO
SaaS 产品 (Notion/Figma)   ⭐⭐⭐⭐⭐  应用化体验
协作编辑 / 在线表格         ⭐⭐⭐⭐⭐  长时间停留 + 实时更新
单团队的工具类应用          ⭐⭐⭐⭐    控制力强、状态可控
登录后的产品主体           ⭐⭐⭐⭐    一登进去就是一个"应用"
```

## 二、MPA 更合适的场景（你公司订单类页面就在这边）
```bash
特征                                     说明
─────────────────────────────────────────────────────────────────
① 多调用方进入同一页面                   邮件 / 短信 / App / 其他业务都能直达
② 多团队协作维护                         不同团队负责不同 .html
③ 业务流程一次性                         取消订单 / 修改信息 / 查看凭证
④ SEO 重要 / 内容型                      博客、新闻、产品介绍页
⑤ 嵌入 App webview                       native 监听 URL 切换
⑥ 历史 URL 包袱重                        老用户书签、邮件链接、SEO 收录
```

*典型 MPA 项目*：
```bash
携程多页 CSR (CRN-Web)     ⭐⭐⭐⭐⭐  多团队 + 多调用方 + App 嵌入
内容型网站 (新闻/博客)     ⭐⭐⭐⭐⭐  SEO 第一、首屏快、交互少
电商商品详情页              ⭐⭐⭐⭐    SEO + 多入口 + 一次性流程
政府 / 银行 / 表单系统      ⭐⭐⭐⭐    简单、稳定、SEO 不重要也不亏
```

## 三、混合架构（最常见）：MPA 大壳子 + SPA 局部
现实工程里大部分项目都是**混合的**：

```bash
架构层                              典型实现
─────────────────────────────────────────────────────────────
跨业务/跨页面跳转 (MPA)              location.href / 老 URL 跳转
单业务内部流程 (SPA)                 用 Link / Router 在一个 SPA 内切
渲染方式 (SSR + CSR)                 首屏 SSR + 后续客户端路由

代表方案: Next.js / Nuxt / Remix
携程实践: NFES (基于 Next.js,首屏 SSR + 内部 SPA 跳转)
         xtaro (多页 CSR,跨页 location.href)
```

## 四、SPA vs MPA 选型决策表（直接套用）
按你项目的特征逐项打 √：

```bash
项目特征                                     SPA   MPA
─────────────────────────────────────────────────────────
用户停留时间长 (> 5 分钟)                     ✅    ❌
跨页面要保持状态 (滚动 / 表单 / 播放)          ✅    ❌
单团队维护、紧密关联的几个页面                ✅    ❌
状态复杂、需要全局 store                      ✅    ❌
应用化体验 (像 App, 丝滑过渡)                 ✅    ❌

多团队各管一块                                ❌    ✅
多调用方进入同一页面 (邮件/短信/App)           ❌    ✅
SEO 第一                                     ⚠️    ✅
首屏速度 / 内容型站                           ⚠️    ✅
嵌入 native App webview                       ❌    ✅
历史老 URL 多, 需兼容                         ❌    ✅
业务流程一次性 (办完即走)                      ❌    ✅

打勾多的一边就是适合的方案
打平 → 用混合方案 (Next.js / Nuxt + 多入口)
```

## 五、SPA 不是免费的：用了要付的代价
即使场景适合 SPA，也要意识到这些**额外维护成本**，不然容易写出"半 SPA 半 MPA"的烂代码：

```bash
代价                                                     如何处理
─────────────────────────────────────────────────────────────────────
全局副作用累积 (监听/定时器/DOM)                          严格 useEffect cleanup
SDK 单例初始化竞争                                       _app.tsx 集中管理 init
跨页面状态污染                                           合理设计全局 store 边界
首屏白屏                                                 配 SSR (变 Next.js)
SEO 差                                                   配 SSR
埋点 PV 不会自动触发                                     监听 router.events 手动上报
跳转后 useEffect 错时机                                  注意区分 mount vs 更新
```

## 六、判断"这一次跳转该用 SPA 还是 MPA"的实操问句
代码里写跳转时，三问：

```bash
Q1. 跳过去的页面是不是【你团队维护】的?
    是 → 可以 Link
    不是 → location.href

Q2. 跳过去的页面会不会被【很多其他入口】调用?
    会 → location.href (保证它在所有入口下表现一致)
    不会 → Link 可以

Q3. 跳过去的页面【依赖】当前页的全局 state 吗?
    依赖 → Link (但要小心管理 state)
    不依赖 → location.href (更简单)

3 个答案都倾向 Link, 才用 Link
否则默认 location.href 是更安全的
```

## 七、跨架构对照速记
```bash
架构          代表项目                 跳转方式               适合场景
─────────────────────────────────────────────────────────────────────────
纯 SPA         Vite + React Router      <Link>                单团队后台、应用类
纯 MPA         传统 PHP / WordPress     <a href>              内容型网站、表单系统
SSR + SPA      Next.js / Remix          首屏 SSR + Link       内容 + 应用混合
多页 CSR       携程 CRN-Web / xtaro     location.href 跨页    多团队 + 多调用方
SSG            Astro / Hugo             a href / 部分 hydrate  纯静态内容站
混合 SSR        Next.js + 多 entry       Link 内 + href 跨     大型企业应用
```

## 八、回到最初的核心结论
```bash
SPA 适合的场景 = "应用化体验" + "单团队/紧密关联" + "状态共享"
                  ↓
其他场景默认走 MPA 或混合架构更稳
                  ↓
"用框架" 不等于 "项目就是 SPA":
   决定 SPA 与否的是【你跳转代码用 Link 还是 location.href】
   多团队 + 多调用方场景用 location.href 反而更安全
```
